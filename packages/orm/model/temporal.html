簡介

時序模型比它聽起來的更不可怕。

通常一個資料庫實體代表資料表中的一列，當它被更新時，舊資訊會被覆寫。時序模型允許資料以時間被引用，它使得在一個所給時間查詢一個實體狀態變為可能。

例如，假設你希望追蹤產品的變更，如此當一個訂單交易成功你可以知道產品的狀態而無須重複訂單資料表中的資料。你可以讓產品有時序並且使用訂單時間來引用當時下訂產品的狀態，而非它們目前的狀態，不使用時序資料的情況下就可能會是當下的狀態。

時序模型也可被用在審核變更的事物像 wiki 頁面。任何變更會被自動記錄，而不須使用一張獨立的 log 資料表。

「實體」可能是你設計中的任何項目，例如產品、使用者、多媒體文件夾、任何你會使用做為模型的可被認為是一個實體。時序模型能讓你透過儲存變更，來追蹤這些實體隨著時間推移的變化，每一組新的變更被稱為一個「修訂版本」。因為每一個修訂版本與時間（時序）資料一起儲存，這使得取得一個實體在所給任何時間的狀態變為可能。

這個時序模型的實現在相同資料表儲存實體的修訂版本做為原始資料，不再需要重複的 "log" 資料表。雖然這裡的實現不是實現修訂版本的唯一方式，它是在 ORM 所給的限制之下最彈性的方式，而且事實上，它必須在多種資料庫環境下運作。

為什麼在 ORM 中實現而不是選擇一個允許該功能的資料庫系統？

在 ORM 中實現時序功能能讓技術被抽象出資料庫層，這能讓 ORM 支援的任何資料庫使用時序模型，提供更大的靈活性。此模型不會取代使用支援時序資訊的資料庫使用能力，但提供一個「開箱即用」的實現。

使用

資料表必須被設置一個實體 ID 的複合主鍵，一個標準的自動遞增數字（例如），以及代表有效時間跨度的時間戳記。

class Model_MyTemporal extends Orm\Model_Temporal
{
	protected static $_primary_key = array('id', 'temporal_start', 'temporal_end');
	protected static $_temporal = array(
		'start_column' => 'temporal_start', // 包含此筆資料有效起始時間的行名稱
		'end_column' => 'temporal_end', // 包含此筆資料有效結束時間的行名稱
		'mysql_timestamp' => false, // 如果設為 true 將使用 MySQL 時間戳記，否則是 UNIX 時間戳記
	);
}

請注意，主鍵應該是一個實體 ID 以及時序時間戳記的複合鍵。

任何使用時序模型定義的關聯應該只涉及 ID，而不是 ID 和時間戳記。

delete($cascade = null, $use_transaction = false)

此方法將從資料庫刪除物件。它與一般模型的刪除功能運作的方式完全相同，除了資訊不會從資料庫被移除。這意味著你仍然可以引用該物件在所給時間，但它就目前而言不再有效。

靜態 否
範例 
Model_Product::find(5)->delete();

串聯刪除將正常刪除。任何不是 Soft 或 Temporal 的模型也會正常刪除。

overwrite($cascade = null, $use_transaction = false)

能讓資訊被儲存而不用建立一個新的修訂版本。與 Model::save() 的運作相同。

靜態 否
範例 
$product = Model_Product::find(5);
$product->name = 'Super Awesome 12000';
// Product 會更新而不會在資料庫中建立新的一筆資料。
$product->overwrite();

restore()

如果一個實體已經被刪除，此方法還原該實體到目前狀態。如果該實體沒有被刪除將不會發生什麼事。

靜態 否
範例 
Model_Product::find_revision(5, '2012-11-09 12:04:00')->restore();

purge()

從資料庫移除此實體的所有修訂版本。這是永久的！它不能被撤消。

靜態 否
範例 
Model_Product::find(5)->purge();

這不能被撤消，它將從資料庫刪除實體的所有修訂版本。如果你這樣做資料將被摧毀！
