故障排除

Fuel 支援各式各樣的配置，有時為了正常工作需要一些配置。以下列出了幾個當你在你的機器上設置 Fuel 時可能跳出的情況。

當你的問題沒有列在下面，或已列出但沒有提供適當的解決方案時，請在議題追蹤（http://dev.fuelphp.com）回報此問題。

  • 純淨安裝下 500 錯誤
  • 純淨安裝下語法錯誤
  • 能運作，但首頁是 404。
  • Oil 無法連到 DB 但應用程式可以
  • ORM/DB 回傳我的資料爲 HTML 實體
  • Orm\Exception [ Error ]: Listing columns not failed
  • 在變更 index.php 的位置後，Asset 不運作。
  • 在我變更一個符號連結後，核心、應用程式或套件無法被找到
  • Exception thrown without a stack frame

純淨安裝下 500 錯誤

發生在……

當你的 HTTP 伺服器權限不足以在檔案系統中跟隨符號連結。

解法

檢查 .htaccess 檔案並修改下列行

Option +FollowSymLinks

以便它讀取：

Option +SymLinksIfOwnerMatch

純淨安裝下語法錯誤

Parse error: syntax error, unexpected T_FUNCTION, expecting ')' in /Users/info/Sites/fuel/public/index.php on line 9

發生在……

當在低於 5.3 的 PHP 版本運行 Fuel 時。

解法

在一個分離的測試檔案檢查 phpinfo(); 並看看你是否正在運行 5.3 或更高版本。更多資訊在需求。

當在本地開發時，更新你的伺服器到 PHP 5.3 的最新版本。如果你不是在本地開發，你將需要詢問你的主機供應商。PHP 5.3 以下的版本不只我們不支援，還有 PHP 它們自己本身也是。

我在 autoloader.php 得到 "child pid [xxxx] exit signal Bus error"

發生在……

在未知的情況下，當使用多位元組支援。

解法

這似乎是一個 PHP 的 bug，在撰寫本文的時候還沒有修正被發布。 大多數使用 PHP 5.3.3 和多位元組支援的人沒有回報此問題。 我們還沒有看到使用 PHP 5.4.x 的使用者遇到相同的錯誤。

做為一種變通方法，建議停用 zend-multibyte 和 mbstring，或嘗試使用不同版本的 PHP。

能運作，但首頁是 404。

發生在……

在某些 PHP 安裝必須在 .htaccess 檔案中的 index.php 後面添加一個問號

解法

像這樣添加問號：

# 打開 URL 改寫
RewriteEngine On

# 安裝目錄
RewriteBase /
# 如果 fuel 不在上一層
# RewriteBase /public

RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d

RewriteRule ^(.*)$ index.php?/$1 [L]
# 注意 ? 在 index.php 之後

一些 fcgi 安裝需要此解法的變種：


RewriteRule ^(.*)$ index.php?$1 [L]
# 請注意 ? 後的斜線已經被移除的事實

# 其他可能的設定：

RewriteRule ^(.*)$ /index.php/$1 [L]
# 請注意，前導斜線添加在 index.php 之前

Oil 無法連到 DB 但應用程式可以

mysql_connect(): [2002] No such file or directory (trying to connect via unix:///var/mysql/mysql.sock) in /Users/phil/Sites/fuel/fuel/core/classes/database/mysql/connection.php on 73

發生在……

發生在當你嘗試藉由命令列與資料庫互動，通常在你為你的 MySQL 伺服器使用 MAMP 時。造成這樣的原因是，MAMP 提供的 PHP-CLI 對 MAMP 來說是一個不同的安裝，所以 PHP 對資料庫伺服器一無所知。

注：在執行以下步驟修正問題之前，驗證藉由正常瀏覽器存取的資料庫連線。

解法

在你的 database.php 變更 "localhost" 為 "127.0.0.1"

如果不行：

建立一個指向 MAMP 的 mysql.socket 的符號連結。

$ sudo ln -s /Applications/MAMP/tmp/mysql/mysql.sock /var/mysql/mysql.sock

在某些情況下你必須先建立目錄：

$ sudo mkdir /var/mysql
$ sudo ln -s /Applications/MAMP/tmp/mysql/mysql.sock /var/mysql/mysql.sock

ORM/DB 回傳我的資料爲 HTML 實體

發生在……

Fuel 有安全性的對策被稱為「輸出編碼」，在預設情況下是打開的。 這意味著，任何傳遞給 View 或 Presenter 的值被認為是危險的，除非你明確地標記它為安全的。 所有不安全的值將透過 htmlentities， 包括陣列內容和物件特性。

解法

學習更多關於 View 類別中的安全性。

Orm\Exception [ Error ]: Listing columns not failed

錯誤

Orm\Exception [ Error ]: Listing columns not failed, you have to set the model properties with a static $_properties setting in the model.

發生在……

當使用 ORM 模型不是藉由 MySQL 連線時。

解法

確認 db 配置值是正確的並且測試你的資料庫連線。

更新模型包含的特性。

protected static $_properties = array('id', 'title', 'contents', 'publish');

或在 APPATH/config/db.php 中變更 db 類型為 "mysql"

在變更 index.php 的位置後，Asset 不運作。

發生在……

那是因為你的路徑沒有正確定義，Asset 類別使用一個相對於 index.php 的路徑。

解法

要修正這個問題，你必須修改 app/config/asset.php 去設定正確路徑。注意，在 index.php 中一個名為 DOCROOT 的常數被定義：

define('DOCROOT', __DIR__.DIRECTORY_SEPARATOR);

你可以在 app/config/asset.php 中使用。這將設定路徑回到預設文件夾 DOCROOT/public/assets，子文件夾 css/ img/ 和 js/ 的所在：

'paths' => array(DOCROOT.'public/assets/'),

在我變更一個符號連結後，核心、應用程式或套件無法被找到

發生在……

這可能會發生，因為在 DOCROOT/index.php，realpath() 被用來處理相對路徑，這將快取路徑資訊，包括符號連結。

解法

要修正這個問題，你必須修改 DOCROOT/index.php 並為每一個包含符號連結的路徑使用 clearstatcache() 以清除快取。或者，你可以重載你的網頁伺服器以清除路徑快取。你不能手動發出 clearstatcache() 命令， 因為它被每一個線程快取，且發出該命令將只清除目前線程的快取。

Exception thrown without a stack frame

發生在……

造成此訊息的原因是當 PHP 已經在它的拆解階段（tear-down phase）發生了錯誤，而且沒有錯誤處理程序是可用的了。

解法

為什麼會發生這種情況的原因有很多。我們會嘗試記錄最顯而易見的：

  • 你的網頁伺服器沒有寫入儲存 log 檔案資料夾的權限（預設是 APPPATH.'logs'）。
